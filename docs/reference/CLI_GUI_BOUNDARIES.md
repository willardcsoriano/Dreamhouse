# CLI vs. GUI Boundaries for Salesforce Metadata

## Overview

Not every Trailhead click-path step has a metadata type worth hand-authoring as XML — some are small, stable, and safe to write from scratch or retrieve-then-patch; others are generated by an interactive builder or wizard and are risky to fake. This doc gives a concrete, three-part test for deciding which is which, grounded in a real case (FlexiPage / Dynamic Forms in Badge 05 Unit 5) rather than a general impression of "some things need the GUI." The short version: hand-author only when the Metadata API reference has a worked example of your _exact_ structure, the XML uses identifiers you'd choose yourself rather than builder-assigned ones, and a structurally valid deploy is actually equivalent to passing whatever check you're trying to satisfy. When any of those three fails, build the artifact once in the tool, retrieve it into source, and treat that retrieved file as the real, CLI-redeployable payload from then on. This is the classification rule referenced by `docs/reference/PIPELINE.md`'s Stage 2 (Polish) and applied throughout the `docs/trails/` unit docs via the `[CLI]` / `[GUI]` tags.

## The Decision Rule

Hand-author a metadata type as `[CLI]` only if **all three** hold:

1. **Salesforce's Metadata API reference has a worked example of your exact structure** — not just the type in general. A `FlexiPage` example using a static Record Detail component doesn't cover a Dynamic-Forms-converted one; they're structurally different.
2. **The XML uses semantic, predictable identifiers you'd choose yourself** — not builder-assigned region UUIDs, auto-generated facet names, or opaque `valueList` structures (the representation Dynamic Forms and other wizard-driven features use from Metadata API v49 onward).
3. **A structurally valid deploy is equivalent to passing the check you actually care about.** For most declarative metadata this is true. For wizard-generated types it usually isn't — a hand-written file can deploy successfully as _valid_ metadata while still not matching what a grader (Trailhead's automated check, a code reviewer, a QA script) is actually looking for: field order, active assignment, specific component type, etc.

Any "no" flips the type to `[GUI]`. That's not a fallback or a workaround — it's the documented, repeatable pattern:

**Build once in the tool → retrieve into source → treat the retrieved file as the real payload from then on.** The click-path authors the artifact correctly exactly once, using Salesforce's own generation logic instead of a guess at it. The `sf project retrieve start` → `sf project deploy start` round trip that follows is what makes it a reproducible CLI artifact for every deploy after that — to a fresh sandbox, a second Trailhead Playground, wherever — without ever touching the browser again for that specific component.

## Worked Example: FlexiPage / Dynamic Forms

This is the case that prompted this doc (Badge 05, Unit 5, `[REQ-5.5.G1.1]`–`[REQ-5.5.G1.5]`).

- **Rule 1 fails:** Salesforce's Metadata API developer guide documents the `FlexiPage` schema (`componentInstance`, `fieldInstance`, `region`/`facet` elements) but provides no worked example of a hand-authored Dynamic Forms page — only pages built from static components.
- **Rule 2 fails:** Since API v49, Dynamic Forms metadata is represented via `valueList` structures whose exact shape and identifier conventions aren't spelled out in the reference; they're what the migration wizard happens to produce.
- **Rule 3 fails hardest:** even a structurally valid hand-written `FlexiPage` could deploy without error and still not satisfy Trailhead's challenge check, which verifies specific field placement and page state that only reliably matches what the wizard itself generates.

Verdict: `[GUI]`. The doc's `G1` section walks the Lightning App Builder click-path (Dynamic Forms migration, field reordering), then `G1.RET` retrieves the resulting `Energy_Audit_Record_Page_for_Sales.flexipage-meta.xml` and redeploys it — that retrieved file, not hand-written XML, is the reproducible artifact.

## Quick Reference

Metadata types encountered so far in this repo's Trailhead units, classified against the rule above:

| Type                                                                    | Classification                | Why                                                                                                                                                        |
| :---------------------------------------------------------------------- | :---------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Custom fields / objects (`CustomField`, `CustomObject`)                 | `[CLI]`                       | Small, stable schema; worked examples exist for every field type used.                                                                                     |
| Compact Layouts (`CompactLayout`)                                       | `[CLI]`                       | Flat, semantic XML — field list plus a label. No builder-assigned structure.                                                                               |
| Profile `fieldPermissions`                                              | `[CLI]`                       | Simple boolean pairs (`readable`/`editable`) keyed by field name.                                                                                          |
| Page Layout `relatedLists`                                              | `[CLI]`                       | Small, well-documented block; safe to retrieve-then-patch.                                                                                                 |
| List Views (`ListView`)                                                 | `[CLI]`                       | Documented schema; the risk is wrong _field identifiers_ (see `docs/REPORT.md` Unit 3 hiccups), not wrong _structure_.                                     |
| FlexiPage — static components                                           | `[CLI]`-leaning, verify first | Has worked examples, but confirm the exact component type against current docs before hand-authoring.                                                      |
| FlexiPage — Dynamic Forms conversion                                    | `[GUI]`                       | See worked example above.                                                                                                                                  |
| FlexiPage — Activation (App/Record-Type/Profile/Form-Factor assignment) | `[GUI]`                       | Wizard-driven; retrieve the `CustomApplication` or `FlexiPage` assignment result instead of guessing it.                                                   |
| Flow (`Flow`)                                                           | `[GUI]`, generally            | Canvas-generated XML with internal connector/node IDs; treat the same as Dynamic Forms unless a specific flow is simple enough to satisfy all three rules. |

When a type isn't on this table yet, run it through the three-part rule above and verify against current Metadata API docs (per the AI-drift rule in the root `CLAUDE.md`) rather than guessing from training data — then add it here.
